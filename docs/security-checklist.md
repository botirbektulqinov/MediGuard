# Security Checklist

This checklist tracks defensive controls implemented or required for MediGuard.

## Authentication

- JWT authentication protects API routes that require identity.
- Access tokens are short-lived.
- Refresh tokens are opaque random values.
- Persisted refresh tokens are hashed.
- Refresh token rotation is atomic.
- Refresh token family reuse revokes the whole family.
- Session records track refresh-token families and move to revoked or expired when the family is no longer usable.
- Security Officers can revoke suspicious sessions.
- Passwords are hashed with bcrypt.
- Failed login attempts are persisted.
- Account lockout is enforced after repeated failed logins.
- Disabled accounts cannot login or refresh tokens.
- Auth endpoints have route-specific rate limits.

## Authorization

- Backend authorization is authoritative.
- RBAC permissions protect privileged workflows.
- Patient ownership checks protect patient portal data.
- Assigned-doctor checks protect doctor patient and visit access.
- Clinic-scoped checks protect staff, clinic configuration, patient, lab, appointment, and queue workflows.
- Security Center endpoints require `security.read` or `security.incident.manage`.
- QA Center endpoints require `qa.read`, `qa.manage`, or `qa.execute`.
- Assigned non-QA users cannot close defects; QA Managers control closure, reopen, retest, and triage status changes.

## Input Validation

- Global DTO validation uses whitelist and unknown-field rejection.
- Route params use UUID parsing where IDs are expected.
- Query filters use DTO validation for audit search.
- Business rules reject invalid appointment status transitions, double booking, and appointment slots outside doctor availability.
- Security incident, finding, QA, and medical workflow inputs validate enum values and text length.

## File Upload Security

- Lab result uploads enforce MIME type, extension, and max size.
- Executable uploads are rejected.
- Stored file names are randomized.
- Original file names are sanitized.
- Internal storage paths are never returned in API responses.
- Attachment downloads re-check ownership, assigned-doctor access, or clinic-scoped lab permission.
- Malware scanning remains a future integration point.

## Logging And Monitoring

- Audit logs cover authentication events, invalid logout tokens, refresh-token reuse, patient profile views, patient contact updates, staff changes, role changes, permission changes, user status changes, appointment/queue changes, visit and lab workflows, file downloads, QA workflows, and Security Center workflows.
- Failed login monitor exposes recent failed attempts.
- Sensitive patient access monitor exposes patient, visit, lab, portal, and file-download audit events.
- Suspicious activity rules detect failed login bursts, disabled-account login attempts, excessive patient views, privilege changes, repeated lab downloads, and patient record access without appointment-context evidence.
- Suspicious activity evaluation creates persistent events with evidence payloads.
- Security incidents and findings are durable and audited.
- Session listings do not expose refresh-token family identifiers to the frontend.

## Secrets Management

- No production secrets are committed.
- `.env` files are ignored.
- `.env.example` contains development-only placeholders.
- Environment variables are validated at API startup.
- Swagger is controlled by `SWAGGER_ENABLED` and defaults off in production.
- Production Swagger startup fails unless `SWAGGER_USERNAME` and `SWAGGER_PASSWORD` are set when `SWAGGER_ENABLED=true`.

## Rate Limiting

- Global API throttling is enabled.
- Auth endpoints have lower route-specific throttles.
- Further per-account and per-IP adaptive throttling is a future enhancement.

## Error Handling

- Business errors use NestJS exceptions.
- Validation errors are generated by framework validation.
- Unauthorized and forbidden responses avoid leaking sensitive implementation details.
- Passwords, access tokens, refresh tokens, secrets, and raw credentials are not logged.
- Request completion logs include method, path, status, latency, and request ID only.

## Production Operations

- Liveness and readiness endpoints are available for platform health checks.
- Backup and restore scripts exist for the production-like Compose stack.
- Production runbook documents TLS, secrets, database hardening, monitoring, and incident response expectations.
- Compliance readiness and penetration test scope are documented separately.

## Future Hardening

- Move refresh tokens from localStorage to HttpOnly Secure SameSite cookies.
- Add immutable audit log retention.
- Add malware scanning for uploaded lab-result files.
- Add notification delivery for high-severity security events.
- Add Playwright E2E coverage for critical browser security flows.
